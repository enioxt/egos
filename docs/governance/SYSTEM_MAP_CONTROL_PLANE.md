## Purpose

Defines the single orchestrator contract for system maps in the EGOS ecosystem: what exists, who owns it, freshness rules, and the one update flow.

---

## Map Inventory

### Machine Map (for LLMs and agents)
- **File:** `egos/docs/SYSTEM_MAP.md`
- **Owner:** EGOS kernel
- **Purpose:** Activation document for LLMs. Points to SSOTs. Contains `<!-- llmrefs -->` block.
- **Freshness rule:** Must be updated whenever a new canonical SSOT is added or removed
- **Who reads it:** LLMs at session start (`/start`), agents during initialization
- **Size limit:** 150 lines max (enforced manually — not in pre-commit currently)

### Human Map (cross-repo global topology)
- **File:** `~/.egos/SYSTEM_MAP.md`
- **Owner:** EGOS kernel (propagated by `governance-sync.sh`)
- **Purpose:** Human-readable view of the full EGOS ecosystem across all repos
- **Freshness rule:** Updated by `bun run governance:sync:exec` after kernel changes
- **Who reads it:** Founder, collaborators, onboarding sessions

### Orchestrator Contract (this file)
- **File:** `egos/docs/governance/SYSTEM_MAP_CONTROL_PLANE.md`
- **Owner:** EGOS kernel
- **Purpose:** Rules for maintaining both maps — one place, no drift
- **Freshness rule:** Updated when a new repo is added to the ecosystem or a map rule changes

## One Update Flow

Any time the ecosystem changes (new repo, new package, removed surface, renamed SSOT), follow exactly this sequence:

```
1. Edit canonical SSOT in egos/
   └── docs/SYSTEM_MAP.md          ← machine map
   └── docs/SSOT_REGISTRY.md       ← if SSOT ownership changed
   └── docs/CAPABILITY_REGISTRY.md ← if capability changed
   └── docs/strategy/ECOSYSTEM_CLASSIFICATION_REGISTRY.md ← if surface class changed

2. Run: bun run governance:sync:exec
   └── Propagates .guarani/, workflows, docs to ~/.egos/

3. Run: bun run governance:check
   └── Verifies no drift between kernel and ~/.egos/

4. Update leaf repo system maps (if applicable)
   └── Each leaf has its own docs/SYSTEM_MAP.md (leaf_local)
   └── Leaf maps reference kernel SSOT but keep domain truth local

5. Record in TASKS.md or handoff doc
   └── One line: what changed and why
```

---

## Freshness Signals

A system map is considered **stale** when:

| Signal | Threshold | Action |
|--------|-----------|--------|
| New repo added to ecosystem, not in map | Immediate | Update SSOT_REGISTRY + machine map |
| Package added to `packages/`, not in CAPABILITY_REGISTRY | Immediate | Update CAPABILITY_REGISTRY |
| Last governance sync > 7 days with kernel changes | 7 days | Run `bun run governance:sync:exec` |
| SYSTEM_MAP.md references a file that no longer exists | On pre-commit | Fix or remove the reference |
| TASKS.md references a doc that no longer exists | On task review | Fix during sprint planning |

## Anti-Patterns (do not do these)

- **Do not create a second machine map.** There is exactly one: `docs/SYSTEM_MAP.md`.
- **Do not edit `~/.egos/SYSTEM_MAP.md` directly.** It is generated by `governance-sync.sh`.
- **Do not add repo topology to `AGENTS.md`.** AGENTS.md is repo identity, not ecosystem map.
- **Do not keep separate system maps per feature branch.** The map lives on main and is updated forward.
